01Infrastructure

What a knowledge base looks like one year in.

Year zero looks like a folder. Year one, with intentional architecture, looks like something you can actually ask questions of. The gap between them is three structural decisions, not a bigger budget.

Convolving Editors June 2025 5 min read Infrastructure
02Year zero: the file dump

Every firm has one. Nobody uses it.

It is usually on SharePoint. Occasionally on Google Drive. Sometimes in a folder on a server that was last touched in 2019. It contains deal memos going back seven years, IC presentation decks in seventeen slightly different templates, board packs for portfolio companies, market studies, management presentations, a surprising number of Excel files with no apparent relationship to anything else, and – if the firm has been around long enough – PDFs that were scanned from paper at some point in the mid-2010s.

Search returns either nothing or seventeen versions of the same document, differentiated only by modification date and whether someone appended "FINAL" or "FINAL_v2" to the filename. Nobody trusts the search. Nobody uses it. When someone needs to find the risk analysis from the last healthcare deal, they send a message in a chat channel asking if anyone remembers where it is. Usually someone does. Sometimes nobody does.

This is not a failure of discipline or organisational hygiene. It is a structural consequence of treating document storage as archiving rather than as memory. The folder is designed to hold files. It was never designed to answer questions. Adding AI to a file dump does not change that. What you get is a model that can search seventeen versions of the same document slightly faster than a human can – which is not the return on investment anyone had in mind.

The firms that have useful knowledge infrastructure a year in did not get there by adding more storage or better tagging conventions. They got there by making three decisions about structure before they started ingesting documents.

03The three structural decisions

Not more budget. Three decisions made before ingestion starts.

1. What is the unit of retrieval?

The default answer is "the document." It is the wrong answer. A document is a container. What the team actually needs to retrieve is a section, a specific data point, a named decision, a flagged risk. When the unit of retrieval is the document, a query about healthcare deal risks returns thirty-page PDFs. When the unit of retrieval is the named-risk entry from section four of a deal memo, the same query returns a structured list that can be read in ninety seconds.

Deciding on the unit of retrieval means deciding how documents get chunked before they enter the corpus. It means deciding which sections of a deal memo are worth indexing separately, which are navigational boilerplate that can be stripped, and which represent genuinely distinct pieces of knowledge. This is editorial work. It requires someone with domain knowledge – someone who understands what an analyst will actually ask – to make decisions about structure. It cannot be delegated to a chunking algorithm.

2. What metadata does each unit carry?

Retrieval without metadata is guessing. A corpus of two hundred deal memos, each chunked into sections, is still nearly useless if none of the sections carry information about which deal they came from, when the deal was active, what stage it was at, who authored the analysis, and what type of document it originated in.

The minimum metadata set for an investment firm's deal corpus is: deal name, deal date (or vintage), document type (IC memo, management presentation, board pack), document date, section type (risk analysis, market sizing, management assessment), and author or team. With these six fields attached to every chunk, a query like "risks flagged in healthcare deals from 2022 to 2024" becomes answerable in seconds. Without them, it is a three-day archaeology project that ends with someone printing things out.

Most firms underestimate how much of this metadata already exists in their systems – in file names, folder structures, email threads, and deal management tools. The extraction is not trivial, but it is not a clean-sheet problem either. The hard part is deciding on the schema before ingestion, not retrofitting it later.

3. Who is responsible for ingestion?

This is the decision that kills the most knowledge base projects. If ingestion is nobody's job, nothing current ever gets added. The corpus reflects the firm as it was at the moment of initial build, not the firm as it is now. Within six months, the gap between "what is in the knowledge base" and "what the team actually knows" has grown wide enough to make the tool unreliable. Analysts learn not to trust it. Partners stop asking for it. The infrastructure atrophies.

Ingestion needs a named owner, a defined cadence (after each IC meeting, after each deal is closed, after each quarterly board pack), and a process that is simple enough to follow without reminders. In most firms, this means building the ingestion step into an existing workflow – the same process that files the IC deck in the shared drive also runs the extraction and tagging that adds it to the knowledge corpus. The marginal effort needs to be close to zero, or compliance will drift toward zero over time.

The most common mistake: treating knowledge architecture as an IT project. It is an editorial project. The question is not "where do files live." It is "what does the team need to remember, and in what form."
04What changes

Queries replace archaeology. New hires get institutional memory in hours.

When the three decisions are made well, the shift in day-to-day experience is significant enough that the team notices within weeks, not months. "What were the three main risks flagged on the last three healthcare deals" becomes a query that takes forty seconds, not a three-day archaeology project that requires tracking down whoever wrote the memos and hoping they remember the context.

The onboarding case is the starkest. A new principal who joins a firm with a well-structured knowledge corpus can, within a few hours of focused query work, develop a working understanding of the firm's deal history, the patterns in how it evaluates risk, the sectors where it has the deepest institutional view, and the specific decisions where it has changed its mind over time. That same information, in a file dump, is essentially inaccessible – not because it does not exist, but because the structure required to surface it does not exist.

Deal team efficiency on second-look processes compounds over time. When a sector comes back around, the analyst is not starting from scratch. The previous analysis is retrievable at the level of specific claims, specific risk assessments, specific market numbers – not at the level of "here is a seventy-page PDF from 2021."

05What does not change

The knowledge base does not think. It remembers.

This distinction is worth being direct about, because the hype around retrieval-augmented systems tends to obscure it. The knowledge base does not write memos. It does not generate investment theses. It does not replace the judgment that comes from pattern recognition across deals, from reading a management team across multiple interactions, from understanding what a market actually feels like to operate in.

What it does is surface the right context at the right time. When an analyst is building a view on a new opportunity in a sector the firm has looked at twice before, the knowledge base gets them to the relevant precedent in minutes rather than days. The analyst still thinks. The model found the relevant materials – it did not interpret them, it did not weigh them against the current opportunity, and it did not decide what to do with them.

Firms that get the most from knowledge infrastructure are the ones that are clear about this internally. They position the tool as memory, not intelligence. The team uses it more, trusts it more, and maintains it more carefully when they understand what it actually is. The firms that oversell it internally – where the tool is positioned as "AI that knows everything we know" – tend to run into credibility problems the first time it surfaces something incomplete or dated, and adoption collapses.

Year one looks different from year zero not because the system is smarter. It looks different because the structure is right, the metadata is clean, and someone is responsible for keeping it current. That is, ultimately, an organisational decision before it is a technical one.
